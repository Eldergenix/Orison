# Migration guide: 0.1 to 0.2

This document describes the migration from the `0.1.x` series to
`0.2.0`. The 0.2 release is *not yet cut* at the time this guide is
written; the shape of the migration is locked, the details of each
rule (and the precise diagnostic IDs) will be filled in when 0.2 is
tagged. Treat this guide as the *format* of every migration guide that
follows: a stable structure with concrete rules slotted in.

## Scope

`0.1.x` was the bootstrap pre-release series. It established every
artefact tier from `STABILITY.md`, but several surface-syntax
constructs are tagged tier 2 (edition-gated), meaning they may move
between 0.1 and 0.2 with automated migration. The 0.2 release pulls
those constructs into their final shape and introduces the first
binding `edition = "2027.2"` line for downstream packages.

The 0.2 release does **not** change any tier-1 surface. JSON schema
contracts, CLI subcommand names, diagnostic IDs, and the capability
lifecycle stay identical. If you only consume tier-1 artefacts, the
upgrade is a no-op.

## Edition transition mechanics

The edition machinery is documented in `STABILITY.md` section 1
(Tier 2). Briefly:

- Every package declares its edition in `ori.toml`:
  ```toml
  [package]
  edition = "2027.1"
  ```
- The compiler reads the edition before parsing and selects the
  parser rules for that edition.
- Two packages on different editions can interoperate, because the
  compiled artefacts (capsules, signatures, JSON envelopes) are
  edition-agnostic — only the surface syntax differs.
- Edition transitions are migration-tool-assisted; they never require
  hand editing.

The 0.2 release introduces edition `2027.2`. To upgrade a package:

1. Run `ori migrate --from 0.1 --to 0.2 --dry-run` to preview every
   change.
2. Drop `--dry-run` to apply.
3. Edit `ori.toml` to bump `edition = "2027.2"`.
4. Run `ori check --json` on every module. Expected: clean.
5. Commit.

The migration tool is *additive only*: it never deletes user code and
never reorders declarations. Every change it makes is one of:

- Rewriting a deprecated keyword to its new form.
- Rewriting a deprecated parameter syntax.
- Adding a missing return type that the new edition requires.

## Required code changes

Until 0.2 ships, the concrete list of rules is illustrative. The
**format** of each rule is fixed and will appear in the released
guide as:

### Rule 0.2-R001: example placeholder

**What changed.** A description of the syntax change, with a one-line
before/after.

**Before:**
```
(0.1 syntax shown here)
```

**After:**
```

```

**Affected diagnostic.** The diagnostic ID that fires on un-migrated
source under the new edition (typically a `W` warning during the
deprecation window, promoted to an `E` error after one minor
release).

**Migration tool action.** What `ori migrate` does automatically.

**Manual fallback.** What you do if you cannot run the tool (e.g.
vendored source you cannot regenerate).

### Rule 0.2-R002: example placeholder

(Same shape as R001.)

When 0.2 ships, the rules table is expected to contain three to five
entries — the bar for an edition-gated change is high, and most
changes will continue to ship within an edition as additive minor
features.

## The migration tool

The tool itself is already in the bootstrap:

```bash
ori migrate --from 0.1 --to 0.2 --dry-run [path]
```

Flags:

| Flag                   | Meaning                                                  |
|------------------------|----------------------------------------------------------|
| `--from <X>`           | Source language version, dotted (e.g. `0.1`).            |
| `--to <Y>`             | Target language version, dotted (e.g. `0.2`).            |
| `--dry-run`            | Compute and print the plan; do not write any file.       |
| `--json`               | Emit `ori.migration_report.v1` instead of human text.    |
| `[path]`               | Package directory or single file. Defaults to `.`.       |

The `ori.migration_report.v1` envelope shape:

```json
{
  "schema":      "ori.migration_report.v1",
  "from":        "0.1",
  "to":          "0.2",
  "files":       [
    {
      "path":         "src/api.ori",
      "rules_fired": ["0.2-R001", "0.2-R003"],
      "operations":  [
        {
          "op":     "change_signature",
          "target": "sym:store.api.get_products",
          "before": "fn get_products() uses http, db.read",
          "after":  "fn get_products() -> List[Product] uses http, db.read"
        }
      ],
      "diagnostics": []
    }
  ],
  "summary": {
    "files_changed":  1,
    "files_examined": 4,
    "rules_fired":    2
  }
}
```

Every operation is a Patch IR op (see
[recipe 03](../cookbook/03-agent-driven-refactor.md)). The migration
tool is built on top of Patch IR, which means three properties hold:

- **Auditable.** The full operation list is in the envelope; you can
  diff it against `git status` after the apply to confirm nothing
  unexpected happened.
- **Reversible.** Run `ori migrate --from 0.2 --to 0.1 --dry-run` and
  the tool emits the inverse operations. (Inverse migration is best-
  effort: if 0.2 added a construct that has no 0.1 equivalent, the
  tool refuses with `MIG0001` and names the offending site.)
- **Idempotent.** Running the migration twice on the same source is a
  no-op; the second run reports `rules_fired: []`.

## Rollback procedure

If a migration goes wrong, the recovery path is:

1. **Use git.** The migration tool only writes files. If you ran it
   inside a clean working tree (which the tool warns about if you
   did not), `git restore .` returns you to the pre-migration state.
2. **Reverse migrate.** If you have already committed and need to
   stay on 0.1 a little longer, run
   `ori migrate --from 0.2 --to 0.1` to walk the rules backward.
   This works for every rule whose forward path is mechanical; rules
   that are inherently one-way (e.g. a deprecated alias that is
   removed in 0.2) cannot be reversed and the tool emits `MIG0001`.
3. **Pin the toolchain.** `ori.toml` can pin a specific toolchain
   version. If your package depends on 0.1-only features that 0.2
   removed, pin `[toolchain].version = "0.1"` until you can rework
   the dependency.

A failed migration never leaves the source in a half-converted state:
the tool buffers every operation in memory, validates the entire
plan, and only then writes files. A failure at any point aborts the
plan and leaves the source identical to the pre-run state.

## What is not in 0.2

The 0.2 release does not:

- Change any JSON envelope schema (those are tier 1).
- Change any diagnostic ID (those are tier 1).
- Remove any CLI subcommand or flag (those are tier 1).
- Change the borrow rules or the effect propagation rules (those are
  semantic, not surface, and would require a major bump).

If you find a 0.1 -> 0.2 change that violates one of those promises,
file an issue tagged `stability-break`; it is a bug in either the
release or this guide.

## When to upgrade

The 0.1 series will receive patch releases (security and bug fixes)
for **at least two minor releases after 0.2 ships** — that is, until
0.4 lands. There is no urgency to upgrade unless you want a feature
0.2 introduces. The migration tool keeps working for the full
deprecation window.

The full deprecation policy lives in `STABILITY.md` section 6
(deprecation policy, binding at 1.0): every deprecated surface emits
a warning for at least two minor releases, then is removed in a
major release with a migration guide.

## Reading the rest of the release notes

When 0.2 ships, the canonical release notes live in `CHANGELOG.md`
under the `0.2.0` heading. Each rule named in this guide will link to
the corresponding `CHANGELOG.md` entry, which in turn links to the
PR that introduced the change. The chain is:

```
docs/migration/0.1-to-0.2.md   <-- this guide
       |
       v
   CHANGELOG.md   (rule by rule)
       |
       v
       PR
       |
       v
   diagnostic ID + new docs
```

This chain is the same for every future migration guide. The format
of this file is the template for `0.2-to-0.3.md`, `0.3-to-0.4.md`,
and the 0.x-to-1.0 guide.
