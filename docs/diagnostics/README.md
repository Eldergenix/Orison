# Orison Diagnostics

This directory contains the machine-generated index of every stable
diagnostic identifier emitted by the Orison toolchain. The companion file
[`INDEX.md`](./INDEX.md) is the authoritative inventory of IDs, their
families, the first source location that defines them, and a one-line
context message. It is produced by `scripts/build_diag_index.py` and
verified on every CI run by `crates/ori-compiler/tests/diag_index_freshness.rs`.

## ID-prefix taxonomy

Every Orison diagnostic is keyed by a short, opaque identifier. The first
character (or short alphabetic word) identifies the *family* — the
subsystem that owns the diagnostic and the broad meaning of the message.
Identifiers are stable: once published they may not be repurposed or
silently retired. The current families are:

| Prefix    | Owner                          | Meaning                                              |
| --------- | ------------------------------ | ---------------------------------------------------- |
| `E`       | parser, resolver, type checker | hard errors that prevent compilation                 |
| `W`       | linter, symbol pass            | warnings that do not block compilation               |
| `B`       | borrow checker                 | ownership and lifetime violations                    |
| `S`       | string lexer                   | string literal and Unicode lexing failures           |
| `N`       | numeric lexer                  | numeric literal lexing and overflow                  |
| `A`       | async runtime                  | scheduler and future-leak diagnostics                |
| `Q`       | SQL/static analysis            | query type and shape checks                          |
| `D`       | design tokens                  | UI design-token consistency warnings                 |
| `R`       | router, runtime                | HTTP route and runtime arity/type errors             |
| `CAP`     | capability runtime             | capability denial, policy, and delegation errors     |
| `RND`     | UI renderer                    | renderer tree/shape validation                       |
| `DSK`     | desktop bundler                | desktop packaging and platform diagnostics           |
| `PUB`     | publishing pipeline            | registry publish-time failures                       |
| `MOB`     | mobile bundler                 | mobile permission and platform diagnostics           |
| `PRE`     | preprocessor                   | reserved for build-time preprocessing diagnostics    |
| `PROTO_E` | protocol                       | reserved for protocol negotiation errors             |
| `AUD`     | workspace audit                | dependency, capability, and security audit findings  |

The numeric part is a four-digit code allocated within the family. By
convention `0001-0099` covers the "core" rules of a family, `0100-0999`
covers second-tier rules added during stabilization, and `1000+` covers
later additions and experimental gates.

## Adding a new diagnostic

1. Pick a family that already covers the concern. Do **not** invent a
   new family without a matching entry in the table above; if you need
   one, open an RFC under [`docs/rfcs`](../rfcs).
2. Allocate the next free number inside the family. The freshness test
   will surface conflicts as drift in `INDEX.md`.
3. Emit the diagnostic with a stable, English message and an
   actionable agent summary. Pair the ID with documentation under
   [`docs/compiler/diagnostics/`](../compiler/diagnostics/) when the rule
   warrants an explanation longer than one line.
4. Regenerate the index with
   `python3.13 scripts/build_diag_index.py`. Commit both the source
   change and the updated `INDEX.md`.

## Regenerating the index

The index is regenerated by a stdlib-only Python 3.13 script. From the
repository root:

```sh
python3.13 scripts/build_diag_index.py          # write docs/diagnostics/INDEX.md
python3.13 scripts/build_diag_index.py --check  # exit non-zero on drift (used by CI)
```

The script enumerates every `crates/*/src/*.rs` file, identifies each
diagnostic ID literal, and selects the most informative call site
(preferring the canonical builder over test assertions and constant
arrays). The output is fully deterministic: families and IDs sort
lexically, message strings are normalised to a single line, and source
links are relative so the file diffs cleanly across machines.

The Rust integration test
[`crates/ori-compiler/tests/diag_index_freshness.rs`](../../crates/ori-compiler/tests/diag_index_freshness.rs)
calls the script with `--check` and fails if the committed
`INDEX.md` does not reflect the current crate sources. The test runs as
part of `cargo test --release -p ori-compiler` and is also exercised by
the workspace-wide validation gate.

If you cannot run Python 3.13 locally, run `make validate` (or the
equivalent CI workflow) to regenerate the file in a clean environment.
