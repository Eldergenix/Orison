# Orison 1.0 Release Plan

This document is the **binding plan** for the Orison 1.0 release. It
defines what must be true before 1.0 is cut, which gates the release
must pass, how the sequencing works from RC1 to final, what the
public-facing communication looks like, and how governance changes
the moment the tag lands.

Everything in this document is normative: a checked box is a
release-blocking commitment, not an aspiration. If a box is unchecked
when the time comes to cut RC1, the release does not happen.

---

## 1. Pre-1.0 checklist

Every tier-1 surface (per [`STABILITY.md` §1](../STABILITY.md)) must
be **stable and documented** with its binding contract explicitly
listed. This is the contract that makes 1.0 *the* 1.0 — every
artefact in this checklist becomes immutable within the 1.x series.

### 1.1 JSON schema contracts

Every shipping `schemas/*.schema.json` file must:

- [ ] Have its `$id` declared (no draft suffix).
- [ ] Be listed in `docs/compiler/REGISTRY_PROTOCOL.md` or the per-
      command tutorial that produces it.
- [ ] Have at least one round-trip test in `tests/` that produces a
      sample envelope and validates it against the schema.

Schemas in scope (current count: 30+, frozen at RC1):

| Family | Schema |
|--------|--------|
| Diagnostics | `ori.diagnostic.v1` |
| Source structure | `ori.capsule.v1`, `ori.agent_map.v1`, `ori.symbol_card.v1` |
| Patch IR | `ori.patch.v1`, `ori.patch_check.v1`, `ori.patch_apply.v1`, `ori.patch_explain.v1` |
| Build and run | `ori.run.v1`, `ori.build_report.v1`, `ori.benchmark.v1`, `ori.doctor.v1` |
| Surfaces | `ori.openapi_report.v1`, `ori.ui_manifest.v1`, `ori.wasm_component.v1`, `ori.capability.v1`, `ori.capability_runtime.v1`, `ori.db_check.v1`, `ori.migration_graph.v1`, `ori.design_tokens_report.v1` |
| Package and supply chain | `ori.package_check.v1`, `ori.lockfile.v1`, `ori.audit_report.v1`, `ori.sbom.v1`, `ori.provenance.v1`, `ori.publish_receipt.v1`, `ori.registry_list.v1`, `ori.sandbox_result.v1` |
| Agent | `ori.agent_diagnose.v1`, `ori.agent_tests.v1`, `ori.agent_changed.v1`, `ori.agent_symbol_list.v1`, `ori.model_loop_telemetry.v1` |
| Imports | `ori.graphql_import.v1`, `ori.rpc_import.v1`, `ori.preprocess.v1` |
| Migration | `ori.migration_report.v1` |

The binding contract: from 1.0, none of these schemas changes its
field types, field names, or required-vs-optional status. Additive
changes (new optional fields) are minor. Anything else is a v2
schema published alongside the v1.

### 1.2 CLI surface

Every subcommand listed by `ori --help` must:

- [ ] Have a stable name, a stable flag set, and a stable exit-code
      mapping documented in `docs/tutorial/CHEATSHEET.md`.
- [ ] Have a `--json` mode that emits the contract envelope from
      §1.1.
- [ ] Have a deterministic exit code (0 = ok, 1 = error-level
      diagnostic, 2 = usage error).

Subcommand count at RC1 target: 27+. Each one is locked: no
renaming, no flag removal, no exit-code remap inside the 1.x series.
Deprecation of a flag is allowed (warning for >= 2 minor releases
before removal in a major), but the deprecation never breaks the
flag's behaviour.

### 1.3 Language keywords

Every item-introducing keyword (`module`, `import`, `fn`, `type`,
`service`, `view`, `actor`, `query`, `migration`, `capability`) and
every body keyword (`let`, `return`, `if`, `else`, `match`, `for`,
`in`, `lambda`, `try`, `Some`, `None`, `Ok`, `Err`, `Unit`) plus the
type forms (`wraps`, the primitives, the parameterised generics) is
frozen. Adding a new keyword that would break existing source is a
major bump gated on an edition transition (per
[`STABILITY.md` §1, Tier 2](../STABILITY.md)).

### 1.4 Diagnostic IDs

The full diagnostic index in `docs/diagnostics/INDEX.md` must:

- [ ] Be regenerated by `scripts/build_diag_index.py` at RC1 time.
- [ ] Have every ID linked to its emitting source location.
- [ ] List every documented prefix family (`E`, `W`, `B`, `S`, `N`,
      `A`, `Q`, `D`, `R`, `CAP`, `RND`, `DSK`, `PUB`, `MOB`, `AUD`,
      `P`, `MIG`).

The contract: each ID is permanent. `E0100` stays `E0100`.
Promoting a warning to an error is breaking (gated on edition).
Demoting an error to a warning is minor (existing valid source stays
valid).

### 1.5 Public Rust types

The public Rust API exposed by `ori-compiler` to consumers (agent
libraries, third-party tooling) is tier 1 from 1.0:

- `ori_compiler::ast::*`
- `ori_compiler::diagnostic::*`
- `ori_compiler::source::*`
- `ori_compiler::patch::*`

Adding new pub items is minor. Renaming, removing, or changing the
signature of an existing pub item is a major bump.

### 1.6 Patch IR operation taxonomy

The op set listed in `schemas/patch.schema.json` is frozen:
`insert_node`, `replace_node`, `delete_node`, `change_signature`,
`insert_match_arm`. Adding a new op is minor; removing or changing
semantics of an existing op is breaking.

### 1.7 Capability lifecycle

The capability lifecycle in `SECURITY.md` (the rules for
`[capabilities].declared`, `uses` clauses, audit propagation,
runtime enforcement, and the `CAP****` diagnostic family) is frozen
from 1.0. The capability runtime types
(`CapabilityToken`, `CapabilitySet`, `CallContext`, `GuardOutcome`,
`AuditEntry`, `DelegationChainToken`, `RevocationList`) and the
guard entry points (`guard_call`, `guard_call_at`,
`guard_call_with_audit`, `guard_call_with_chain`) are tier-1 stable.

## 2. Required gates

Each gate must be green at RC1, RC2, and final. The gates are run by
`scripts/validate_all.py --full` and the cross-platform CI workflow
at `.github/workflows/cross-platform-verify.yml`.

### 2.1 validate_all green on all three OSes

`python3.13 scripts/validate_all.py --full` must exit 0 on:

- **Ubuntu LTS** (the canonical Linux target)
- **macOS** (latest stable; both x86_64 and arm64)
- **Windows** (Windows Server 2022 in CI; Windows 11 spot-checks)

The `--full` mode includes static checks (layout, JSON contracts,
shell hooks), the test suite, schema validation, and the bench
sanity run.

### 2.2 Conformance suite green

The conformance suite under `tests/conformance/` (golden fixtures
per command + per envelope) must pass on every supported toolchain.
A conformance failure on any platform blocks the release.

### 2.3 Benchmark regression < 10%

The bench harness (`ori bench --json`) must report **no more than
10% regression** on any of its 32 suites relative to the published
`BENCHMARKS.results.json` baseline. The 10% gate is measured on the
median of 32 samples; outliers are excluded by the harness's
default trimming.

A regression that exceeds the gate is a release-blocker. The
maintainer who introduced it has two options: fix the regression or
get RFC approval to accept it.

### 2.4 No outstanding `stability-break` issues

Any issue tagged `stability-break` in the repository must be closed
or explicitly downgraded before RC1. The label is reserved for
violations of `STABILITY.md`; an open issue with this label is a
hard blocker.

### 2.5 Cookbook and tutorial parity

Every code block in `docs/tutorial/`, `docs/cookbook/`, and
`docs/migration/` must be extractable and pass `ori check --json`
without errors. The extraction script is part of
`scripts/validate_all.py`; a doc that drifts from the toolchain is a
release-blocker.

## 3. Release sequencing

The release is cut in three named milestones: **RC1**, **RC2**, and
**1.0.0 final**. Each milestone is tagged, published as a pre-release
on GitHub, and announced to the early-adopter Slack and the
mailing list. The 1.0.0 final is the only one that goes to the
broader channels listed in §4.

### 3.1 RC1

- **Tag:** `1.0.0-rc.1`.
- **Window:** at least **14 days** before RC2.
- **What it contains:** every tier-1 surface frozen; all gates green.
- **What changes during the window:** only bug fixes, no new
  features. A feature merged after RC1 starts the clock again
  (i.e. the project re-cuts RC1 with a new suffix).
- **Outcome:** the release goes to RC2 if no `release-blocker` issues
  are opened during the window. If a blocker is opened, it must be
  closed before the window restarts.

### 3.2 RC2

- **Tag:** `1.0.0-rc.2`.
- **Window:** at least **7 days** before final.
- **What it contains:** RC1 plus only blocker fixes.
- **What changes during the window:** documentation, release notes,
  packaging. No code changes outside `docs/`, `CHANGELOG.md`, and
  release-engineering artefacts.
- **Outcome:** final if no new blockers. Otherwise back to RC2
  (re-cut with new patch suffix).

A pattern that has worked in other languages is: if RC2 needs more
than one re-cut, the project promotes the current state to RC3 and
extends the window by another 7 days. This is acceptable practice
under this plan.

### 3.3 1.0.0 final

- **Tag:** `1.0.0`.
- **Cut from:** the head of the RC2 branch at the end of the RC2
  window.
- **Distribution:** GitHub release with SBOM (every supported format),
  signed Docker image, Homebrew formula update, registry publish.
- **Communication:** the full plan in §4 runs on the day of the tag.

Re-cutting 1.0.0 after the tag has been published is not permitted.
A post-1.0 fix ships as 1.0.1.

## 4. Communication plan

The release announcement runs on the day the `1.0.0` tag is pushed
and the artefacts are signed. Every channel below has a named owner
and a pre-approved message template.

### 4.1 Release notes

- Location: `CHANGELOG.md` entry under `1.0.0` heading, plus the
  GitHub release body.
- Length: 800-1500 words. Sections: highlights, breaking changes
  from 0.x (linking the migration guide), tier-1 lockdown
  announcement, gratitude / contributor list, what's next.
- Owner: the release captain (rotates per release; for 1.0 named in
  `MAINTAINERS.md`).
- Pre-flight: peer-reviewed by at least two other maintainers, with
  a 48-hour comment window.

### 4.2 Blog post

- Location: `https://orison-lang.org/blog/orison-1-0`.
- Length: 1200-2500 words. Targeted at readers who have heard of
  Orison but not yet tried it.
- Structure: the elevator pitch, three end-to-end use cases (REST
  API, wasm component, agent-driven refactor), the stability
  commitment, where to start.
- Owner: the release captain plus a designated writer (the writer
  drafts; the captain reviews for technical accuracy).
- Pre-flight: shared in draft with the maintainers list one week
  before the tag.

### 4.3 Hacker News

- Submission title format: `Orison 1.0 (orison-lang.org)`.
- Submission URL: the blog post.
- Submission time: same day as the tag, between 13:00 and 15:00 UTC
  (post-Asia working hours, peak North-American attention).
- Owner: a community member with a long-lived HN account submits;
  the release captain is *not* the submitter (HN frowns on
  self-promotional submission patterns from project leads).
- Comment moderation: at least two maintainers spend the first six
  hours after submission answering questions on the thread. The
  comment guide lives in `docs/RELEASE-1.0-comment-guide.md`
  (created at RC2).

### 4.4 Lobste.rs

- Submission title: `Orison 1.0` with tags `programming` and
  `release`.
- Submission URL: the blog post.
- Submission time: same day as HN, but after the HN submission
  (lobste.rs links to its HN counterpart automatically).
- Owner: a maintainer with a lobste.rs invite; falls back to a
  community member if no maintainer has one.

### 4.5 Other channels

- **Mastodon**: post from the project account
  (`@orison@hachyderm.io` or equivalent); link the blog post; use
  the hashtag `#orisonlang`.
- **r/programming, r/rust**: posted only if a community member
  volunteers; not driven by the project.
- **Mailing list / newsletter**: full release notes go to the
  early-adopter list within 24 hours of the tag.

The discipline is: every channel post is approved in advance and
attributed to a named owner. The plan exists so the release week is
not chaotic.

## 5. Post-1.0 governance shift

The day the `1.0.0` tag lands, the contribution model changes. Pre-
1.0, changes to tier-1 surfaces ship via maintainer review
(documented in `GOVERNANCE.md`). Post-1.0, every change to a tier-1
surface goes through the **RFC process**.

### 5.1 What requires an RFC

Any change that affects a tier-1 artefact, including:

- Adding, removing, or renaming a CLI subcommand or flag.
- Adding, removing, or renaming a JSON schema field.
- Adding or removing a language keyword (only at edition boundaries).
- Adding a new diagnostic ID family.
- Changing the Patch IR op taxonomy.
- Changing the capability lifecycle.
- Changing the deprecation policy itself.

Additive changes (new schema, new flag, new diagnostic ID *within an
existing family*) are minor and follow the existing GOVERNANCE
process. Substantive changes (anything in the list above) are
RFC-only.

### 5.2 RFC process summary

The full process lives in `docs/rfcs/README.md` (introduced as part
of the GOVERNANCE refresh shipping with 1.0). The skeleton:

1. **Pre-RFC discussion** in the issues or forum. Anyone may open.
2. **RFC draft** as a PR adding a numbered file under `docs/rfcs/`.
3. **Comment window** of at least 21 days for substantive RFCs.
4. **Decision** by the steering committee (named in
   `MAINTAINERS.md`), recorded in the RFC as `accepted`,
   `rejected`, or `postponed`.
5. **Implementation** by the RFC author or a designated maintainer.
6. **Release** in the next minor or major version, with the RFC
   number cited in the changelog.

The bar for an accepted RFC is rough consensus *and* working code:
the RFC may not land without a reference implementation that passes
every gate from §2.

### 5.3 What does not require an RFC

- Bug fixes that preserve behaviour.
- Performance work that does not change observable behaviour.
- Documentation edits.
- Tests.
- Tier-3 (experimental) and tier-4 (internal) changes.

The intent is to keep the RFC bar high enough that day-to-day work
flows naturally, while ensuring that any change that touches the
1.0 contract is deliberate and recorded.

### 5.4 The steering committee

The steering committee is the body that adjudicates RFCs. Its
composition, replacement procedure, and decision-making rules live
in `GOVERNANCE.md`. For 1.0, the committee is the set of
maintainers listed in `MAINTAINERS.md` as of the RC2 cut.

The committee meets monthly to review open RFCs, prioritise blocking
work, and produce a public meeting summary. Anyone may attend; only
committee members vote.

## 6. The 1.0 contract in one sentence

From the moment the `1.0.0` tag is pushed:

> *Every artefact in this document is binding. Breaking changes to
> tier-1 surfaces require a major version bump and a migration
> guide. Tier-2 surfaces evolve through edition transitions with
> automated migration. Tier-3 and tier-4 surfaces continue to
> evolve as needed.*

That sentence is the promise the project makes to every
1.0-adopter. Every line of this document exists to make it
honourable.

## 7. Open questions tracked as RFCs

These items are explicitly *out of scope* for 1.0 and are tracked
as post-1.0 RFCs:

- The Stage 1 self-hosted compiler bring-up. Design done; tracked
  under RFC-001 (placeholder; opens after 1.0 cut).
- The production capability runtime persistence (M35 wave 3).
  Tracked under RFC-002.
- The full GC story for wasm-32 GC. Tracked under RFC-003.
- The async actor system completion. Tracked under RFC-004.

None of these is required for 1.0. Their completion is target for
the 1.x series under the RFC process.
