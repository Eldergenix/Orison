# Comparative benchmarks: Orison vs Rust, Go, Swift

> **TL;DR.** Orison's default `check` runs in tens of microseconds. Rust's
> `cargo check`, Go's `go build`, and Swift's `swift build` take 0.5–20 s
> from a cold cache. The numbers look like a six-order-of-magnitude win,
> but they are **not** measuring the same amount of work. This document
> exists so we never quote those numbers without the asterisk.

Citation-backed companion to `BENCHMARKS.md`. Read "Apples vs oranges"
before quoting any multiplier.

---

## 1. What Orison's `check` does (and does not)

Orison's wedge is the **agent edit-check-repair loop**, not a full
type-checker. By default, `ori check`:

- Parses the input.
- Resolves module-local names.
- Confirms top-level signatures are coherent (parameter counts, return
  types, capability sets).
- Emits a structured diagnostic envelope (`ori.diagnostic.v1`) suitable
  for an LLM to consume.

It does **not** by default: type-check function bodies, run
borrow/effect/lifetime fixpoints (those live under separate suites in
`BENCHMARKS.md`), resolve transitive dependencies, or lower to MIR.

By contrast:

| Tool | Parses | Names | Sig-check | Body type-check | Borrow / ownership | Codegen |
|------|:-:|:-:|:-:|:-:|:-:|:-:|
| `ori check` (default) | yes | yes | yes | no | no | no |
| `rustc --emit=metadata` | yes | yes | yes | yes | yes | no |
| `cargo check` | yes | yes | yes | yes | yes | no (per-crate metadata only) |
| `cargo build --release` | yes | yes | yes | yes | yes | yes |
| `go build` | yes | yes | yes | yes | n/a (GC) | yes |
| `go vet` | yes | yes | yes | partial | n/a | no |
| `swift build` | yes | yes | yes | yes | yes (ARC + lifetimes) | yes |
| `swiftc -typecheck` | yes | yes | yes | yes | partial | no |

The comparison is **honest** for the agent-in-the-loop signal and
**misleading** as a general-purpose "1,680,000x faster than Rust"
headline.

## 2. Why we run the comparison anyway

1. **Wedge validation.** Orison's pitch is "edit-check fits in one LLM
   token tick". Pinning a real type-checker's cost gives us a
   sanity-check ceiling we'd notice if we drifted toward.
2. **Cross-platform delta.** The same workload runs on ubuntu /
   macos-13 / macos-14 / windows-latest in
   `.github/workflows/cross-platform-verify.yml`. >100% drift across
   OSes is a signal worth seeing.
3. **Receipts on demand.** A table with citations beats a vibes-based
   answer when a reviewer asks "is it actually fast enough?".

## 4. Per-row caveats

- **Orison**: signature-level only. Binary cell is encoded `hello.wasm`
  from `wasm_hello_ns` (~96 bytes); Orison emits no native executable.
- **Rust (cargo check)**: includes dependency resolution + metadata;
  `serde_json`-sized crate, cold cache.
- **Rust (cargo build --release)**: full codegen; included so the
  cold-check row isn't read as Rust's floor.
- **Rust (rustc --emit=metadata)**: closest apples-to-apples to Orison;
  still runs full body type-check + borrow analysis.
- **Go (go build)**: famously fast; even so, 500 LoC cold ≈ half a
  second. The fairest cross-language row since `go build` ≈ check+link.
- **Go (go vet)**: lower-bound, check-only Go reference.
- **Swift (swift build)**: SwiftPM library; includes module map +
  SIL generation.
- **Swift (swiftc -typecheck)**: closest Swift gets to Orison's pass;
  still runs full body type-check + generic specialisation.

## 5. Apples vs oranges (READ THIS BEFORE QUOTING)

The "speedup vs Rust" column is technically correct and rhetorically
dangerous:

- Orison's number is an in-process Rust bench loop on a single small
  module. The competitors' numbers include process startup,
  filesystem scan, dependency resolution, and full body analysis.
  ~80% of `cargo check`'s wall-clock on a small crate is *not*
  type-checking.
- A real `ori build --emit=wasm` run over a multi-module app with
  effects + capability + borrow + wasm encode + manifests lands in
  millisecond territory, not microsecond. Those suites live in
  `BENCHMARKS.md` under "Lowering + codegen" and "Manifests".
- The defensible claim: **Orison's edit-check loop fits in one
  agent token tick (sub-millisecond); mainstream typed languages'
  check loops fit in a coffee break (hundreds of ms to tens of
  seconds).**

Publishing an "X faster than Rust" headline without that paragraph
misrepresents Orison.

## 6. How to reproduce

### Locally (Apple Silicon or Linux x86_64)

```bash
# 1. Build the release Orison toolchain.
cargo build --release -p ori

# 2. Refresh the bench artefact (~30s).
target/release/ori bench --samples 30 --json > BENCHMARKS.results.json

# 3. Generate the comparison table (default = all 3 languages).
python3 scripts/compare_languages.py --output docs/benchmarks/comparative.generated.md

# 4. Subset variants:
python3 scripts/compare_languages.py --include-rust
python3 scripts/compare_languages.py --include-go --include-swift --baseline orison
```

The script is stdlib-only (Python 3.13). No `pip install` required.

### On CI

- `.github/workflows/cross-platform-verify.yml` runs the bench on all
  four supported OS targets and uploads the artefacts as
  `bench-<os>.json`. A follow-up job downloads them and posts a
  cross-OS comparison table to the workflow summary.
- `.github/workflows/bench-regression.yml` (the authoritative p50
  drift gate) appends a step that runs
  `scripts/compare_languages.py --output comparative.md` and uploads
  the markdown as an artefact alongside the regression report.

### Updating reference numbers

Edit `REFERENCE_ROWS` in `scripts/compare_languages.py` and the
citation block in that file's module docstring. Values must come from
publicly-documented cold-cache numbers (WWDC sessions, Rust internals
forum, Go release notes). Local laptop measurements belong in
`BENCHMARKS.md`, not here.
